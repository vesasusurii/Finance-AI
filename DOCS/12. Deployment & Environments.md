# Deployment & Environments — Borek Finance Invoice Automation

> **Principle:** Same environment for every developer on the team. Docker Compose is the default for local work. Production uses the same service boundaries with stricter secrets, HTTPS, and backups.

---

## 1. Overview

| Concern | Implementation |
|---|---|
| Local / team dev | **Docker Compose** — PostgreSQL + backend (+ frontend when scaffolded) |
| Application code | `backend/` (FastAPI), `frontend/` (React + Vite) — see docs 4 and 6 |
| Database | PostgreSQL 16 in Compose; managed Postgres in production |
| File storage | Docker volume `upload_data` → `/data/uploads` in backend container |
| External APIs | **OpenAI only** (invoice extraction). No bank API. |
| Auth | JWT in **httpOnly** cookie — see **5. API Specification 1.md** |
| Health checks | `GET /api/health`, `GET /api/ready` — see **5. API Specification 1.md** |

## 2. Environments

| Environment | Purpose | Who uses it | Data |

| **local** | Daily development on laptops | All developers | Disposable Postgres volume; test uploads |

## 3. Docker Compose Services (Starter)

Default stack in repo root `docker-compose.yml`:

```
┌─────────────────────────────────────────────────────────────┐
│  docker compose (local)                                      │
│                                                              │
│  ┌──────────────┐     ┌──────────────┐     ┌─────────────┐ │
│  │     db       │     │   backend    │     │  frontend   │ │
│  │  Postgres 16 │◄────│  FastAPI     │     │  Vite dev   │ │
│  │  :5432       │     │  :8000       │     │  :5173      │ │
│  └──────────────┘     └──────┬───────┘     └──────┬──────┘ │
│         ▲                    │                    │        │
│         │              upload_data volume          │        │
│         │              (invoices, bank Excel)      │        │
│         └──────────────────────────────────────────┘        │
│                    (frontend: profile `full` when ready)     │
└─────────────────────────────────────────────────────────────┘
```

| Service | Image / build | Host port | Notes |
|---|---|---|---|
| `db` | `postgres:16-alpine` | `5432` | Persistent volume `postgres_data` |
| `backend` | `build: ./backend` | `8000` | Hot-reload mount in dev |
| `frontend` | `build: ./frontend` | `5173` | **Profile `full`** — enable after `frontend/package.json` exists |

**Starter state:** `backend` includes a minimal `main.py` with `/api/health` and `/api/ready` so the team can verify Compose before full app code lands. Replace/extend as you implement doc 6.

## 4b. Production Compose profile

Local production-like stack (no `--reload`, Redis password, nginx TLS):

```bash
# One-time: self-signed certs for https://localhost
sh infra/nginx/generate-dev-certs.sh

# Set in .env: ENVIRONMENT=production, REDIS_PASSWORD, COOKIE_SECURE=true, strong JWT_SECRET

docker compose -f docker-compose.yml -f docker-compose.production.yml --profile production up -d --build
```

| Service | Notes |
|---|---|
| `nginx` | TLS on :443, proxies `/api` → backend, `/` → `frontend-prod` |
| `frontend-prod` | Vite production build (`frontend/Dockerfile.prod`) |
| `db` / `redis` | No host ports published |
| `backend` / `worker` | No source bind mount; `uvicorn` without `--reload` |

**Backups:** For Compose Postgres, schedule `pg_dump` from the host or use managed Postgres (Supabase) backups. Example cron:

```bash
docker compose exec -T db pg_dump -U finance finance_ai | gzip > backup-$(date +%F).sql.gz
```

## 6. Environment Variables

Copy `.env.example` to `.env` in the **repo root**. Compose passes these to `backend` (and `db` where noted).

### Database

| Variable | Example (local) | Required | Description |
|---|---|---|---|
| `POSTGRES_USER` | `finance` | Yes | Postgres role |
| `POSTGRES_PASSWORD` | *(strong random)* | Yes | Postgres password |
| `POSTGRES_DB` | `finance_ai` | Yes | Database name |
| `DATABASE_URL` | `postgresql+asyncpg://finance:pass@db:5432/finance_ai` | Yes | SQLAlchemy URL for backend — host **`db`** inside Compose |

**Important:** From your laptop (outside Docker), use `localhost` instead of `db`:

`postgresql+asyncpg://finance:pass@localhost:5432/finance_ai`

### Application

| Variable | Example | Required | Description |
|---|---|---|---|
| `JWT_SECRET` | *(32+ char random)* | Yes | Signs `access_token` cookie |
| `JWT_EXPIRE_MINUTES` | `480` | No | Session length (8h default) |
| `ENVIRONMENT` | `local` | Yes | `local` \| `staging` \| `production` |
| `CORS_ORIGINS` | `http://localhost:5173` | Yes | Comma-separated frontend origins |
| `STORAGE_PATH` | `/data/uploads` | Yes | Invoice PDFs/images, bank Excel — maps to volume |
| `OPENAI_API_KEY` | `sk-...` | Phase 1+ | Invoice extraction |
| `OPENAI_MODEL` | `gpt-4o-mini` | No | Extraction model |
| `LOG_LEVEL` | `info` | No | `debug` in local only if needed |

### Frontend (host or container)

| Variable | Example | Required | Description |
|---|---|---|---|
| `VITE_API_BASE_URL` | `http://localhost:8000` | Yes (frontend) | Browser calls this base for `/api/*` |

### Cookie / security (production)

| Variable | Example | Required | Description |
|---|---|---|---|
| `COOKIE_SECURE` | `true` | Prod only | `Set-Cookie` Secure flag |
| `COOKIE_SAMESITE` | `lax` | No | Match **5. API Specification** |

In **local** Docker, `COOKIE_SECURE=false` so cookies work over `http://localhost`.

## 7. First-Time Local Setup (Team)

### Step 1 — Clone and configure

```bash
git clone <repo-url>
cd Finance-AI
cp .env.example .env
```

Edit `.env`: set `POSTGRES_PASSWORD`, `JWT_SECRET`, and later `OPENAI_API_KEY`.

### Step 2 — Start database and backend

```bash
docker compose up -d --build
```

Verify:

```bash
curl http://localhost:8000/api/health
# {"status":"ok"}

curl http://localhost:8000/api/ready
# {"status":"ready","db":"ok"}  (after DB is up and backend connects)
```

### Step 3 — Database migrations (when Alembic exists)

```bash
docker compose exec backend alembic upgrade head
```

Run after every pull that includes new migrations under `backend/db/migrations/versions/`.

### Step 4 — Users

Create accounts in the app (**Admin → Users**) or rely on users already in the database. Store credentials in the team password manager — not in Git.

### Step 5 — Frontend (when scaffolded)

```bash
docker compose --profile full up -d --build
```

Open `http://localhost:5173`. Login flow per **4. Frontend Architecture** and **5. API Specification**.

**Until frontend exists:** run only `db` + `backend`, or use API via curl/Postman.

## 9. Networking & URLs (Local)

| Consumer | URL |
|---|---|
| Browser → frontend | `http://localhost:5173` |
| Browser → API | `http://localhost:8000/api/...` |
| Backend → Postgres | `db:5432` (Docker network) |
| Health (load balancer / ops) | `http://localhost:8000/api/health`, `/api/ready` |

### CORS

Backend `CORS_ORIGINS` must include the exact frontend origin (scheme + host + port). For local:

```
CORS_ORIGINS=http://localhost:5173
```

Multiple origins (staging + local):

```
CORS_ORIGINS=http://localhost:5173,https://finance-staging.borek.internal
```

### Vite proxy (optional alternative)

Instead of `VITE_API_BASE_URL=http://localhost:8000`, frontend `vite.config.ts` can proxy `/api` → `http://backend:8000` **inside** Compose network. Document one approach per team and stick to it.

## 10. File Storage (Uploads)

Financial files **must not** be committed to Git.

| Item | Location |
|---|---|
| Compose volume | `upload_data` → `/data/uploads` in backend |
| Env | `STORAGE_PATH=/data/uploads` |
| Contents | Invoice PDFs, JPG/PNG, bank statement `.xlsx` |

**Backup (production):** snapshot volume or sync to encrypted object storage (S3-compatible). **Retention:** keep originals for audit per ODT security section — align with `uploaded_files` rows in **7. DB Schema**.

**Permissions:** backend process user only; no public HTTP path to raw files.

## 11. Database Operations

### Migrations (Alembic)

Workflow matches **7. DB Schema**:

```
1. Change SQLAlchemy model in backend/models/
2. docker compose exec backend alembic revision --autogenerate -m "description"
3. Review generated SQL in versions/
4. docker compose exec backend alembic upgrade head
5. Commit migration file to Git
```

**Team rule:** pull → `alembic upgrade head` before starting work.

### Backups

| Environment | Approach |
|---|---|
| local | Optional — volume wipe acceptable |
| staging | Nightly dump `pg_dump` |
| production | Automated daily backup + test restore monthly |

```bash
# Example manual dump (production — run on DB host or sidecar)
pg_dump -h <host> -U finance -d finance_ai -Fc -f finance_ai_$(date +%Y%m%d).dump
```

### Restore (staging drill)

```bash
pg_restore -h <host> -U finance -d finance_ai --clean finance_ai_YYYYMMDD.dump
```

## 12. Backend Container

### Dockerfile (starter)

- Base: `python:3.12-slim`
- Installs `requirements.txt`
- Exposes `8000`
- Dev command: `uvicorn main:app --host 0.0.0.0 --port 8000 --reload`

### When full app lands (doc 6)

- `config.py` reads env via `pydantic-settings`
- Startup: SQLAlchemy engine, `httpx` client, ensure `STORAGE_PATH` exists
- Shutdown: dispose engine, close HTTP client

### Production command (no reload)

```bash
uvicorn main:app --host 0.0.0.0 --port 8000 --workers 2
```

Increase workers only after async profile testing; MVP can use 1 worker.

### Profile `full`

Frontend service is behind Compose profile **`full`** until `frontend/package.json` exists.

### Dev

- Port `5173`

## 14. Staging & Production Deployment

Docker Compose is for **local/team**. Production options (pick one; document choice in README when decided):

| Platform | Backend | Frontend | DB | Uploads |
|---|---|---|---|---|
| **VPS + Compose** | Same images, no `--reload` | nginx `dist` | Postgres container or managed | Volume or S3 |
| **Railway / Render** | Dockerfile deploy | Static site service | Managed Postgres addon | Platform disk or S3 |
| **Split managed** | Container service | CDN/static | Neon / RDS | S3 |

### Production checklist

| Item | Requirement |
|---|---|
| HTTPS | TLS on all routes; `COOKIE_SECURE=true` |
| Secrets | Platform secret manager — not `.env` in image |
| `ENVIRONMENT` | `production` |
| `CORS_ORIGINS` | Production frontend URL only |
| `JWT_SECRET` | Unique, rotated on compromise |
| `OPENAI_API_KEY` | Org key with usage limits |
| DB | Managed Postgres, SSL, backups |
| Uploads | Encrypted at rest |
| Health | LB checks `/api/health` and `/api/ready` |
| Logging | Structured logs; no invoice content in debug logs |
| Access | VPN or IP allowlist if internal-only |

**No bank API** in any environment — uploaded Excel only (**3. Technical Workflow**).

## 16. Security & Compliance (Deployment)

Aligned with ODT security requirements and **1. Project Summary**:

| Rule | Deployment implication |
|---|---|
| No bank API | No bank credentials in env |
| Auth required | All `/api/*` except login/health/ready |
| RBAC | `finance` vs `admin` in JWT claims |
| Audit | `audit_logs` table retained; DB backups include it |
| File access | No public bucket listing; signed URLs if using S3 |
| Secrets | `.env` in `.gitignore`; rotate `JWT_SECRET` if leaked |
| User offboarding | `is_active=false` — do not delete users with audit FK (**7. DB Schema**) |

## 17. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `connection refused` to DB | `db` not healthy | `docker compose ps`; wait for `healthy` |
| Backend `db: unreachable` on `/api/ready` | Wrong `DATABASE_URL` host | Use `@db:` inside Compose, `@localhost:` on host |
| CORS error in browser | `CORS_ORIGINS` mismatch | Match exact `http://localhost:5173` |
| Cookie not set after login | `COOKIE_SECURE=true` on HTTP | Set `COOKIE_SECURE=false` locally |
| Upload fails | `STORAGE_PATH` not writable | Check volume mount `upload_data` |
| Port 5432 in use | Local Postgres running | Stop local Postgres or change host port in compose |
| Frontend profile won't start | No `package.json` | Omit `--profile full` until scaffolded |
| Alembic not found | Backend image old | `docker compose build --no-cache backend` |
