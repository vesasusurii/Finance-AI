# Frontend Architecture — Borek Finance Invoice Automation

> **Principle:** Finance users upload, review, and approve. The UI never auto-confirms financial data on behalf of the user — every uncertain case surfaces for human decision.

## 2. Pages

### Dashboard

Entry point after login. Shows Finance the current state at a glance.

| Element | Data source |
|---|---|
| Pending review count | `GET /api/invoices?review_status=needs_review` |
| Unmatched invoices count | `GET /api/invoices?match_status=unmatched` |
| Recent uploads | `GET /api/invoices?limit=10&sort=created_at` |
| Last reconciliation run | `GET /api/reconciliation/results?limit=1` |

No edits on this page — read-only summary with navigation links to each section.

---

### Upload Invoices

Finance drops one or many invoice files. Each file is processed independently.

**Requirements from project spec:**
- Drag-and-drop area + file picker fallback
- Batch upload (multiple files in one action)
- Per-file status: `Pending → Processing → Processed / Failed`
- Error message if extraction fails for a specific file
- "Review" link per file that goes to the Review Queue for that invoice

**Flow:**
```
User drops files → FileDropzone → multipart POST /api/invoices/upload
  → per-file status polling or WebSocket update
  → on success: link to invoice row in Purchase Invoices Table
  → on failure: error message + Needs Review badge
```

**Key rule:** `paid at (Date)` is not shown or set on this page — that comes from bank matching only.

---

### Upload Bank Statement

Finance uploads the Bank Statement Excel. Preview before committing.

**Requirements:**
- Single Excel file upload
- Preview parsed transaction rows before saving
- Show detected invoice numbers from `Komenti / Comment` per row
- Processing status (row count, failures if any)

**Flow:**
```
User picks Excel → POST /api/bank-statements/upload
  → backend returns parsed preview rows
  → user confirms → rows saved to bank_transactions
  → "Run Matching" button → POST /api/reconciliation/run
```

### Purchase Invoices Table

The main data table. Mirrors the **Purchase Invoices Database** Excel columns.

**Columns shown:**

| Column | Editable | Notes |
|---|---|---|
| Invoice Date | Yes | Finance can correct AI extraction |
| Name of Company | Yes | |
| Adress of Company | Yes | |
| Invoice Number | Yes | Matching key — edit with care |
| Amount | Yes | |
| Account Details | Yes | |
| Internal Note/Description | Yes | |
| Client / Employee Related | Yes | |
| paid at (Date) | **No** | Set only by reconciliation |
| paid by | Yes | Manual by Finance |

**Interactions:**
- Inline edit via `EditableCell` → `PUT /api/invoices/{id}`

**Implemented route:** `/documents` (`routes/documents.tsx`) — table + side drawer on row click.

Shows every row from the uploaded bank statement.

| Credited Amount | Vlera e kredituar |

---

### Matching Results

Summary of the last (or any) reconciliation run.

**Sections:**

| Section | Description |
|---|---|
| Matched | Invoice number found in both DB and bank comment; `paid_at_date` set |
| Unmatched Invoices | In Purchase Invoices DB but no bank comment match found |
| Unmatched Transactions | In bank statement but no invoice number found |
| Needs Review | Comment has a number but no invoice in DB; or multiple candidates |
| Batch Candidates | Comment has multiple numbers — all matched |

**Interactions:**
- Approve match → `POST /api/reconciliation/approve-match`
- Reject match → `POST /api/reconciliation/reject-match`
- Re-run matching → `POST /api/reconciliation/run`

---

### Manual Review Queue

Finance resolves cases the AI or matcher could not handle confidently.

**Each queue item shows:**
- Original invoice file preview (`InvoiceFilePreview`)
- AI-extracted fields (editable)
- Bank transaction candidate (if applicable)
- Reason for review (e.g. `low_confidence`, `no_invoice_in_db`, `missing_amount`)
- Confidence badge

**Actions per item:**
- **Approve** → marks invoice as `approved`; match as `approved` if bank candidate present
- **Reject** → removes match candidate; keeps invoice for manual handling
- **Save correction** → `PUT /api/invoices/{id}` with edited fields + re-approve

---

### Export

Download the updated Purchase Invoices Database as Excel.

**Flow:**
```
User sets optional filters (date range, category, match status)
  → GET /api/export/purchase-invoices-excel
  → browser downloads .xlsx file
```

The exported file has **exactly** the official Finance column structure. System-only fields (`extraction_confidence`, `match_status`, `review_status`) are excluded unless Finance explicitly approves adding them.

## 3. Communication Rules

### Call Direction

```
Page → Hook → api/ → FastAPI backend
```

- Pages never call `fetch` directly — they use hooks
- Hooks call `api/` functions — which contain all URLs and headers
- `api/` files use native `fetch()` — no SDK, no axios

```typescript
// api/invoices.ts — example shape
export async function uploadInvoices(files: File[]): Promise<UploadResponse> {
  const form = new FormData();
  files.forEach(f => form.append("files", f));
  const res = await fetch("/api/invoices/upload", { method: "POST", body: form });
  if (!res.ok) throw new Error(await res.text());
  return res.json();
}

// hooks/useInvoices.ts — example shape
export function useInvoices(filters: InvoiceFilters) {
  const [invoices, setInvoices] = useState<Invoice[]>([]);
  const [loading, setLoading] = useState(true);
  useEffect(() => {
    listInvoices(filters).then(setInvoices).finally(() => setLoading(false));
  }, [filters]);
  return { invoices, loading };
}
```

### State

No global state library for MVP. React local state + hooks is enough for this scale. If complexity grows across pages, add React Context or a lightweight library (e.g. Zustand).

| State type | Where it lives |
|---|---|
| Auth (current user + role) | `AuthContext` — wraps entire app |
| Page data (invoices, transactions) | Local state inside each hook |
| Form edits (inline cell edits) | Local `useState` inside `EditableCell` |
| Upload progress (per file) | Local state inside `UploadInvoicesPage` |

## 4. Auth

```
/login → LoginPage → POST /api/auth/login → JWT stored in httpOnly cookie
  → ProtectedRoute checks AuthContext
  → redirect to /dashboard on success
  → redirect to /login if unauthenticated
```

- All routes under `/` are protected — `ProtectedRoute` enforces it
- Role (`finance` or `admin`) from JWT payload controls which actions are visible
- Admin sees all pages; Finance sees all pages except user management (if added later)

## 6. Layer Descriptions

| Layer | What it does | What it does NOT do |
|---|---|---|
| **pages/** | Renders the screen, composes components, passes data to hooks | No direct `fetch` calls |
| **hooks/** | Fetches data, manages loading/error state, exposes refetch | No JSX |
| **api/** | Builds request, calls `fetch`, throws on non-OK response | No state, no JSX |
| **components/** | Reusable visual elements | No data fetching, no business logic |
| **types/** | TypeScript interfaces matching backend schemas | No logic |
| **auth/** | Session check, role guard, login form | No business logic |

## 8. Implementation Patterns

### New page checklist

**`types/` interface → `api/` function → `hooks/` hook → `pages/` component → add route in `App.tsx`**

### File upload (multipart)

```typescript
// Always multipart/form-data — never JSON — for file uploads
const form = new FormData();
files.forEach(f => form.append("files", f));
await fetch("/api/invoices/upload", { method: "POST", body: form });
```

### Inline edit

```typescript
// EditableCell: local state for draft, PUT on blur/enter
const [draft, setDraft] = useState(value);
const save = () => updateInvoice(invoiceId, { [field]: draft });
```

### Confidence badge color rule

| Score | Color | Label |
|---|---|---|
| ≥ 0.90 | Green | Auto-saved |
| 0.70 – 0.89 | Yellow | Needs Review |
| < 0.70 | Red | Manual Required |

### Status badge values

| Domain | Values |
|---|---|
| `processing_status` | `pending`, `processing`, `processed`, `failed` |
| `review_status` | `pending`, `approved`, `needs_review` |
| `match_status` | `unmatched`, `matched`, `needs_review` |
| `reconciliation_status` | `pending`, `matched`, `partial`, `needs_review` |
