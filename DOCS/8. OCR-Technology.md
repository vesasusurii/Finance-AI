# OCR & Invoice Extraction Technology

> **Principle:** OCR reads the document; the LLM structures and validates. The backend owns the final JSON schema — providers are swappable behind one interface.

> **Provider policy:** **OpenAI Vision / GPT is the main OCR technology** for all environments (MVP and production). **Google Document AI** runs **only when OpenAI is unavailable** (outage, timeout, or repeated API errors). Both paths output the same JSON schema.

**Related docs:** [2. Tech Stack.md](./2. Tech Stack.md) · [3. Technical Workflow 1.md](./3. Technical Workflow 1.md) · [9. Matching-Normalization.md](./9. Matching-Normalization.md) · [11. ## Provider Priority (locked)

| Priority | Provider | When used |
|---|---|---|
| **1 — Primary** | **OpenAI Vision** (`gpt-4o-mini` / `gpt-4o`) | Every invoice extraction attempt **starts here** |
| **2 — Fallback** | **Google Document AI** (Invoice Parser) | OpenAI fails after retries (down, 5xx, timeout, rate limit exhaustion) |
| **3 — Local (optional)** | **PDF text** / **Tesseract** | Dev without keys, or both cloud providers unavailable |

Finance workflow does not change between providers — only `provider` in `audit_logs` and optional UI badge (“extracted via fallback”).

## What OCR Does in This Project

OCR (optical character recognition) is **not** the whole invoice pipeline. It is the step that turns pixels or scanned pages into machine-readable text (or layout-aware blocks). A second step — **structured extraction** — maps that content into the fixed JSON schema Finance needs.

```
Invoice file (PDF / image / scan)
        ↓
┌───────────────────┐
│  OCR / Vision     │  ← this document
│  (read document)  │
└─────────┬─────────┘
          ↓ raw text or layout JSON
┌───────────────────┐
│  LLM validation   │  ← OpenAI (primary) — or Google entities on failover
│  + Pydantic       │
└─────────┬─────────┘
          │ (if OpenAI API down)
          └──► Google Document AI → same schema
          ↓
┌───────────────────┐
│  confidence tier  │  → invoices row + review queue
└───────────────────┘
```

**Out of scope for OCR:** bank statement Excel parsing (that is `excel_service`, not OCR). **Never extract** `paid_at_date` or `paid_by` from invoices via OCR.

## Supported Input Types

| Input | OCR approach | Notes |
|---|---|---|
| Digital PDF (text layer) | PDF text extract → **OpenAI** structures JSON; Vision if text empty or garbled | sample_03, sample_04, sample_05 |
| Scanned PDF (image only) | **OpenAI Vision** always (Google if OpenAI down) | sample_01 — `pdfplumber` returns **0 characters** |
| Photo (JPG/PNG) | **OpenAI Vision** (Google if OpenAI down) | Often skewed, shadows — preprocess (see below) |
| Multi-page invoice | Per-page OCR → merge text in page order | Concatenate with `--- Page N ---` markers for LLM |
| Excel invoice | **Phase 2+** — not OCR; direct parse | Optional later |

**MVP formats:** PDF, JPG, JPEG, PNG, scanned PDF.

### Text-layer detection (route to Vision when needed)

```python
MIN_PDF_TEXT_CHARS = 80  # below this → treat as scanned

def needs_vision_ocr(pdf_path: str) -> bool:
    text = extract_pdf_text(pdf_path)  # pdfplumber / pymupdf
    if len(text.strip()) < MIN_PDF_TEXT_CHARS:
        return True  # e.g. sample_01 → 0 chars, 1 embedded image
    # Optional: high replacement char ratio → garbled encoding (sample_02)
    if text.count("\ufffd") / max(len(text), 1) > 0.02:
        return True
    return False
```

| Sample ID | Text chars (pdfplumber) | Route |
|---|---|---|
| `sample_01` | **0** | **Vision only** (scanned) |
| `sample_02` | ~1950 (encoding noise) | Text + Vision or Vision preferred |
| `sample_03` | ~511 | Text extract + OpenAI JSON |
| `sample_04` | ~922 | Text extract + OpenAI JSON |
| `sample_05` | ~1600 | Text extract + OpenAI JSON |

Golden PDFs and expected JSON are maintained outside the repo (or in a local-only folder) — not committed with documentation.

### Per-sample extraction notes

#### `sample_01` — scanned professional invoice (Vision required)

- **Layout:** Bilingual **Invoice / Faturë**; line-item table; single payment block with IBAN + SWIFT.
- **Labels:** `Invoice Ref`, `Invoice Date`, `Date of Taxable Supply`, `Due Date`, `Sub-total / Nëntotal`, `VAT (TVSH)`, `Total Amount Due`, `Payment Reference`.
- **Field targets:**
  - `name_of_company`, `address_of_company` — top issuer block (not bill-to)
  - `invoice_number`: `1/2026/0048` (matches payment reference)
  - `invoice_date`: `2026-01-28` (from `28.01.2026`)
  - `amount`: `1931.78` — use **Total Amount Due**, not sub-total `1637.10`
  - `currency`: `EUR`
  - `account_details`: IBAN `XK051110342170000160` + SWIFT + bank name from payment section
  - `internal_note_description`: line items (e.g. compliance services + admin fee)
- **Normalized `invoice_number`:** `120260048` — see [9. Matching-Normalization.md](./9. Matching-Normalization.md).
- **Risk:** Signature/stamp area — use **`gpt-4o`** if `gpt-4o-mini` misses totals.

#### `sample_02` — trilingual utility invoice

- **Languages:** Albanian + English + Serbian (`FATURA- INVOICE - RACUN`, `Për pagesë / For payment / Za naplatu`).
- **`invoice_number`:** prefer **`159263`** (e-payment platform line); header `Nr.-No.-Br.` may show **`5945698`** — confirm with Finance.
- **`amount`:** use **`Për pagesë` / For payment: `20.60`**, not **`Gjithsej borgii` / Total Due: `49.20`** (includes prior balance `28.60`).
- **`account_details`:** multiple local IBANs — concatenate or first IBAN + “(+N more)”.
- **Tax IDs on document:** `NRF`, `VAT` — not `invoice_number`.
- **OCR risk:** Encoding artifacts — `needs_review` if `confidence_score < 0.85`.

#### `sample_03` — foreign-currency SaaS invoice

- **Labels:** `Invoice number`, `Date of issue`, `Amount due`, `Total`.
- **`invoice_number`:** `3807F638-0011` (strip null / zero-width chars from PDF text).
- **`amount` / `currency`:** `20.00` + **`USD`** — do not default to EUR.
- **`name_of_company`:** issuer in header (US address), not bill-to block.
- **Client ref** in bill-to (e.g. `811915159`) — not `invoice_number`.

#### `sample_04` — Albanian retail / IT

- **Labels:** `Fatura Nr.`, `Data`, `Klienti`, `Për pagesë`, `TVSH`, `NUI`.
- **`invoice_number`:** from `Fatura Nr.` → `10210`
- **`invoice_date`:** `10/02/2026` → `2026-02-10`
- **`amount`:** `198.00` from `Për pagesë` row
- **`internal_note_description`:** product line (e.g. DDR4 RAM module)
- **Footer:** multiple IBANs — same pattern as `sample_02`
- **Note:** `NUI` on PDF may differ from other samples (`811515159` vs `811915159`) — extract as printed; optional `needs_review` on mismatch

#### `sample_05` — subscription invoice (EUR, VAT split)

- **Labels:** `Tax invoice`, `Invoice number`, `Total`, `Date`, `Bill to`, `VAT Registration Number`
- **`invoice_number`:** `132018959018` (long numeric — do not truncate)
- **`invoice_date`:** `04-Feb-26` → `2026-02-04`
- **`amount`:** `54.00` from `Total invoice` (subtotal `45.00` + VAT `9.00`)
- **`client_employee_related`:** contact name in bill-to if present
- **`name_of_company`:** issuer in header (US legal entity + d/b/a line)

## Provider Options

### Comparison

| Criteria | **OpenAI Vision / GPT** (primary) | **Google Document AI** (fallback) | **Tesseract (local)** |
|---|---|---|---|
| Role | **Main OCR** — default for every file | **Backup** when OpenAI is down | Dev / last resort |
| Best for | Varied layouts, photos, Albanian/English mix | Invoice-tuned OCR when OpenAI unavailable | Offline, no API keys |
| Structured fields | Prompt + JSON mode | Native invoice entities → map to our schema | Text only |
| Scanned / photo quality | Strong with good prompts | Strong, invoice-tuned | Weak on poor photos |
| Cost model | Per image/token (normal path) | Per page (only during failover) | Free (CPU) |
| Lock-in | Medium | Medium | Low |
| PHI/finance sensitivity | Data sent to OpenAI | Data sent to Google Cloud | Stays on server |

### Standard stack (MVP + production)

| Step | Stack |
|---|---|
| **Always first** | PDF text extract (if digital PDF) → **OpenAI Vision** → Pydantic validation → confidence tiers |
| **If OpenAI fails** | **Google Document AI Invoice Parser** → map entities → same Pydantic validation (optional OpenAI text pass **only if OpenAI is back** — otherwise rules-only cleanup) |
| **If both fail** | Mark `processing_status = failed` or Tesseract + `needs_review` (no auto-approve) |

## Architecture: Provider Abstraction

All OCR/extraction code should implement one interface. **MVP and production use the same priority:** OpenAI first, Google on failure.

```python
# services/ocr/base.py
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class OcrResult:
    raw_text: str
    provider: str  # "openai_vision" | "document_ai" | "pdf_text" | "tesseract"
    page_count: int
    metadata: dict  # language, warnings, per-page confidence if available

@dataclass
class ExtractionResult:
    fields: dict  # matches InvoiceExtractionSchema
    confidence_score: float
    needs_review: bool
    provider: str
    ocr_result: OcrResult

class InvoiceExtractor(ABC):
    @abstractmethod
    async def extract(self, file_path: str, mime_type: str) -> ExtractionResult:
        ...
```

**Implementations:**

| Class | File (suggested) |
|---|---|
| `PdfTextExtractor` | `services/ocr/pdf_text.py` — no OCR, first pass |
| `OpenAIVisionExtractor` | `services/ocr/openai_vision.py` |
| `DocumentAIExtractor` | `services/ocr/document_ai.py` |
| `TesseractExtractor` | `services/ocr/tesseract.py` — optional |

**Orchestrator:** `invoice_extraction_service.py` always tries primary, then fallback:

```env
OCR_PRIMARY=openai_vision
OCR_FALLBACK=document_ai
EXTRACTION_MODEL=gpt-4o-mini
OPENAI_MAX_RETRIES=2
OPENAI_TIMEOUT_SECONDS=60
```

### Failover logic (implement in `invoice_extraction_service.py`)

```python
async def extract(self, file_path: str, mime_type: str) -> ExtractionResult:
    try:
        return await self._extract_openai(file_path, mime_type)
    except OpenAIUnavailableError as e:
        audit_log("ocr_failover", primary="openai_vision", fallback="document_ai", error=str(e))
        return await self._extract_document_ai(file_path, mime_type)
```

**Treat OpenAI as “down” and switch to Google when any of:**

| Condition | Examples |
|---|---|
| HTTP 5xx | 500, 502, 503 from OpenAI |
| Timeout | No response within `OPENAI_TIMEOUT_SECONDS` |
| Connection error | DNS, TLS, network unreachable |
| Repeated 429 | Rate limit after `OPENAI_MAX_RETRIES` with backoff |
| Invalid API key at runtime | 401 — alert ops; do not silently loop |

**Do not failover for:** bad prompts, validation errors, or unreadable files — those are application errors, not provider outages.

After failover, set `ExtractionResult.provider = "document_ai"` and optionally `needs_review = True` with reason `extracted_via_fallback` so Finance can spot rows processed during an outage.

## Pipeline A — Primary (OpenAI Vision)

```
POST /api/invoices/upload
        ↓
invoice_extraction_service.extract(file)
        ↓
┌─ Try pdf_text (if PDF) ─────────────────────────┐
│  If len(text) > threshold → skip vision           │
└───────────────────────────────────────────────────┘
        ↓ (else)
┌─ OpenAI Vision (PRIMARY) ───────────────────────┐
│  Send page images + system prompt                 │
│  Response: JSON only (strict schema)              │
│  On failure → Pipeline B (Google Document AI)     │
└───────────────────────────────────────────────────┘
        ↓
ai_validation_service.validate(schema)
        ↓
confidence tier → invoices + review_status
```

### Step 1: Preprocess (before API call)

| Step | Tool | Why |
|---|---|---|
| PDF → images | `pdf2image` + Poppler | Vision models need raster pages |
| Max dimension | Resize to ≤ 2048px long edge | Cost + latency |
| Rotation | EXIF auto-rotate (Pillow) | Phone photos |
| Contrast (optional) | Pillow grayscale + autocontrast | Faded scans |
| Password PDF | Reject with clear error | Cannot OCR locked files |

Do **not** over-process — aggressive binarization can hurt Vision models.

### Step 2: OpenAI Vision request

**Models:** `gpt-4o-mini` default; `gpt-4o` for difficult scans or retry after first failure (still OpenAI — not a switch to Google).

**Message shape (conceptual):**

```python
response = await client.chat.completions.create(
    model=os.environ["EXTRACTION_MODEL"],
    response_format={"type": "json_object"},
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {
            "role": "user",
            "content": [
                {"type": "text", "text": "Extract invoice fields from this document."},
                {
                    "type": "image_url",
                    "image_url": {"url": f"data:image/jpeg;base64,{b64_page}"},
                },
            ],
        },
    ],
)
```

Multi-page: either one request with multiple images (watch token limits) or per-page extract + merge in `ai_validation_service`.

### Step 3: System prompt (requirements)

The prompt must enforce:

1. **JSON only** — no markdown, no explanation outside JSON  
2. **Exact field keys** — match `InvoiceExtractionSchema`  
3. **Null/empty** for missing fields — do not invent invoice numbers or amounts  
4. **Currency** — ISO code if visible (e.g. `EUR`)  
5. **Dates** — ISO `YYYY-MM-DD` when possible; if ambiguous, set `needs_review: true`  
6. **Category** — suggest from fixed list (Utilities, Travel, Office, …) — Finance can override  
7. **Exclude** `paid_at_date` and `paid_by` — never return these from invoice OCR  
8. **confidence_score** — 0.0–1.0 self-assessment  
9. **needs_review** — `true` if any critical field uncertain  

**Example system prompt skeleton (tuned for reference samples):**

```text
You are an invoice data extractor for internal finance. The bill-to / Klienti
block is the customer — do not use the customer's NUI/UNI/NRF (e.g. 811915159)
as invoice_number.

Return a single JSON object with these keys only:
invoice_date, name_of_company, address_of_company, invoice_number, amount,
currency, account_details, internal_note_description, client_employee_related,
category, confidence_score, needs_review.

Rules:
- name_of_company = issuer (header), not bill-to / Klienti.
- invoice_number: from "Invoice Ref", "Fatura Nr.", "Invoice number" — not tax IDs.
  Accept formats: 10210, 1/2026/0048, 3807F638-0011, 132018959018, 159263.
- amount: use "Total Amount Due", "Për pagesë", "Amount due", "Total invoice" —
  NOT "Total Due" that includes previous balance (e.g. 49.20 vs 20.60 on utility bills).
- currency: EUR or USD as printed on the document.
- Dates: DD.MM.YYYY, DD/MM/YYYY, DD-MMM-YY, long month names → YYYY-MM-DD.
- Albanian/English/Serbian bilingual labels on same line are common.
- account_details: IBAN (XK...), bank name; multiple banks → concatenate briefly.
- category: Professional services, Utilities, Software, IT / Hardware, Office, Travel, Other.
- Scanned PDFs (no text layer): read tables carefully; total is after VAT line.
- Do not guess invoice_number or amount.
- confidence_score: 0-1; needs_review: true if missing or unclear critical fields.
- Never include paid_at_date or paid_by.
```

### Step 4: Pydantic validation (`ai_validation_service`)

```python
class InvoiceExtractionSchema(BaseModel):
    invoice_date: str | None = None
    name_of_company: str | None = None
    address_of_company: str | None = None
    invoice_number: str | None = None
    amount: float | None = None
    currency: str | None = None
    account_details: str | None = None
    internal_note_description: str | None = None
    client_employee_related: str | None = None
    category: str | None = None
    confidence_score: float = Field(ge=0, le=1)
    needs_review: bool = True
```

**Post-validation rules (code, not LLM):**

| Check | Action |
|---|---|
| `invoice_number` empty | `needs_review = True` |
| `amount` empty or ≤ 0 | `needs_review = True` |
| `confidence_score < 0.70` | `needs_review = True` |
| Date parse fails | `needs_review = True`, `invoice_date = null` |
| Amount string with comma | Normalize `220,66` → `220.66` |

### Step 5: Confidence tiers (align with Technical-Workflow)

| Model `confidence_score` | Backend action |
|---|---|
| ≥ 0.90 | `review_status = approved` (if validation passes) |
| 0.70 – 0.89 | Save row, `review_status = needs_review` |
| &lt; 0.70 | Save row, `review_status = needs_review`, flag “do not finalize” in UI |

Store **`extraction_confidence`** on `invoices` and log provider + model version in **`audit_logs`**.

## Pipeline B — Fallback (Google Document AI)

**Trigger:** OpenAI unavailable after retries (see failover table above). **Not** used as the default path.

```
PDF/image
    ↓
Google Document AI — Invoice Parser (EU region if required)
    ↓
entity map: supplier_name, invoice_id, total_amount, invoice_date, ...
    ↓
Map entities → InvoiceExtractionSchema
    ↓
If OpenAI is reachable again: optional text-only pass for category / Albanian labels
Else: rule-based cleanup only (no second cloud dependency during outage)
    ↓
Pydantic + confidence tiers → DB
    ↓
Flag needs_review if extracted_via_fallback (recommended during outages)
```

### Document AI setup notes

| Item | Recommendation |
|---|---|
| Processor | `INVOICE_PROCESSOR` (pretrained) |
| Region | Choose closest to Kosovo/EU for latency and compliance |
| Input | GCS upload or raw bytes via `documentai.DocumentProcessorServiceClient` |
| Output | `document.entities` — map to internal schema |

**Entity mapping (typical):**

| Document AI entity | Our field |
|---|---|
| `supplier_name` | `name_of_company` |
| `supplier_address` | `address_of_company` |
| `invoice_id` | `invoice_number` |
| `total_amount` | `amount` |
| `invoice_date` | `invoice_date` |
| `net_amount` / `total_tax_amount` | Use `total_amount` for Amount column unless Finance specifies otherwise |
| `iban` / payment fields | `account_details` |

Document AI may not return **category** or **client_employee_related** — fill via OpenAI text-only pass **when OpenAI is back**; otherwise leave empty or use a fixed category list + `needs_review`.

### Fallback confidence

```python
confidence_score = min(
    document_ai_entity_confidence,  # when available
    0.85,  # cap during fallback — prefer human review after outage
)
needs_review = needs_review or provider == "document_ai"  # configurable
```

If Document AI omits `invoice_id` → **force `needs_review`**. Never invent invoice numbers during failover.

## Pipeline C — Local fallback (Tesseract)

Use only when **both** OpenAI and Google are unavailable, or for local dev without API keys.

```
image → Pillow preprocess → pytesseract.image_to_string(lang="eng+sqi")
    ↓
raw_text → manual review queue (do not call OpenAI if it is the reason for failover)
```

| Setting | Value |
|---|---|
| Languages | `eng` + `sqi` (Albanian) if installed |
| PSM | `6` (uniform block) or `4` (single column) — tune per sample invoices |

Tesseract alone is **insufficient for MVP accuracy** on mobile photos; treat as fallback.

## Error Handling & Retries

| Error | `processing_status` | User-facing |
|---|---|---|
| Unsupported MIME | `failed` | “File type not supported” |
| PDF encrypted | `failed` | “Password-protected PDF” |
| OCR timeout | `failed` | “Processing timed out — retry” |
| Invalid JSON from model | retry once, then `failed` | “Extraction failed” |
| Rate limit (429) | retry with backoff on **OpenAI**; then failover to Google | “Busy — retrying…” / “Using backup extractor…” |
| OpenAI down (5xx/timeout) | automatic **Google Document AI** | “Processed with backup OCR” (optional toast) |

**Retry policy:** Max `OPENAI_MAX_RETRIES` on OpenAI per file; second attempt may use `gpt-4o`. If still failing → **Google Document AI** once. If Google fails → `failed` + audit log.

**Audit:** Log `ocr_failed` with `file_id`, provider, error code — never log full card/IBAN in plain text if avoidable.

## Architecture Summary

| Layer | Technology |
|---|---|
| **Primary OCR** | **OpenAI Vision** (`gpt-4o-mini` / `gpt-4o`) |
| **Fallback OCR** | **Google Document AI** when OpenAI is down |
| Structure fields | OpenAI JSON (primary) or Document AI entities → same Pydantic schema |
| Quality gate | `confidence_score` + business rules → `needs_review` |
| Storage | Raw file + extracted JSON + `extraction_confidence` + `provider` on `invoices` |
| Failover | Automatic in `invoice_extraction_service` — logged to `audit_logs` |
| Not OCR | Bank Excel, matching, `paid_at_date` |

## Definition of Done (OCR)

- [ ] Digital PDFs use text extraction before Vision when possible  
- [ ] **Zero-text PDFs** (sample_01) always use **OpenAI Vision** — never mark as “empty file”  
- [ ] Golden set of **5 reference samples** passes eval script thresholds  
- [ ] Scanned PDF and images extract via **OpenAI** with strict JSON schema  
- [ ] OpenAI outage automatically fails over to **Google Document AI** with same schema  
- [ ] Failover events logged; UI or audit shows `provider` (`openai_vision` vs `document_ai`)  
- [ ] Validation enforces required fields and `needs_review` rules  
- [ ] Confidence tiers match Technical-Workflow Step 2  
- [ ] Golden-set eval script runs with documented accuracy targets  
- [ ] Failed extractions set `processing_status = failed` with clear UI message  
- [ ] Audit logs record provider and model version per extraction  
- [ ] `paid_at_date` / `paid_by` never populated by OCR layer
