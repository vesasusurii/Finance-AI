## Overview

PostgreSQL is the **system of record**. The **Purchase Invoices Database** Excel file is an export view — same column names Finance uses today.

All tables are created via **Alembic** migrations from **SQLAlchemy models** in `backend/models/`.

---

## Tables

| Table | Purpose |
|---|---|
| `users` | Finance team accounts and roles |
| `uploaded_files` | Metadata for every invoice PDF/image and bank Excel upload |
| `invoices` | Purchase Invoices Database rows (one per invoice) |
| `bank_statements` | One row per uploaded Bank Statement Excel file |
| `bank_transactions` | Parsed rows from bank Excel (Comment → detected numbers) |
| `invoice_payment_matches` | Links invoice ↔ bank transaction when reconciliation succeeds |
| `review_tasks` | Manual review queue (extraction or match failures) |
| `audit_logs` | Immutable log of extractions, edits, and payment date updates |

**Total: 8 tables**

---

## Relationships

```
users
└── uploads many → uploaded_files

uploaded_files
├── optional source for → invoices (invoice file)
└── optional source for → bank_statements (bank Excel)

bank_statements
└── has many → bank_transactions

invoices
└── has many → invoice_payment_matches

bank_transactions
└── has many → invoice_payment_matches

invoices / bank_transactions
└── may have many → review_tasks (polymorphic target)

audit_logs
└── references user_id + optional entity type/id (no strict FK required for flexibility)
```

### SQL (reference)

```sql
CREATE TABLE invoices (
    id SERIAL PRIMARY KEY,
    invoice_date DATE,
    name_of_company VARCHAR(500),
    address_of_company TEXT,
    invoice_number VARCHAR(200),
    invoice_number_normalized VARCHAR(200),
    amount NUMERIC(18,2),
    currency VARCHAR(10),
    account_details TEXT,
    internal_note_description TEXT,
    client_employee_related VARCHAR(500),
    paid_at_date DATE,
    paid_by VARCHAR(300),
    fixed_status VARCHAR(100),
    category VARCHAR(200),
    extraction_confidence NUMERIC(5,4),
    review_status VARCHAR(50) NOT NULL DEFAULT 'pending',
    match_status VARCHAR(50) NOT NULL DEFAULT 'unmatched',
    source_file_id INTEGER REFERENCES uploaded_files(id),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ix_invoices_invoice_number ON invoices (invoice_number);
CREATE INDEX ix_invoices_invoice_number_normalized ON invoices (invoice_number_normalized);
```

## What We Do NOT Store

| Data | Where | Why |
|---|---|---|
| Live bank credentials | N/A | No bank API — upload only |
| Guessed invoice numbers | N/A | Parser + exact match; else review |
| Auto `paid_by` | N/A | Finance manual rule |
| Full OpenAI prompts per row | Optional in `audit_logs.after` | Keep payloads small; log ids and confidence |

## Normalization Notes (application layer, not DB)

Invoice numbers are normalized in Python before compare. MVP uses **exact match on `invoice_number_normalized`**; `invoice_number` keeps the display value for Excel export.

Examples stored after normalization:

- Invoice row: `26063`
- Bank comment `26.063` → detected `26063` → match

If formats diverge too far, row stays `match_status = needs_review` — no silent wrong payment date.

## Migration Workflow

```
1. Define model in backend/models/
2. alembic revision --autogenerate -m "add invoices table"
3. alembic upgrade head
4. Seed admin user (one-time script)
```

No pgvector extension required for this project.
