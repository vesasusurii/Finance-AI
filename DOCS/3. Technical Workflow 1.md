## End-to-End Workflow (Official Process)

```
1. Finance uploads invoice files (PDF, scan, photo)
2. AI reads each invoice and extracts structured fields
3. Data saved to Purchase Invoices Database (DB → exportable Excel)
4. Finance uploads Bank Statement Excel
5. System reads each transaction row
6. System parses Komenti / Comment for invoice numbers
7. System matches numbers to Invoice Number column
8. On match: Bank Statement Data / Date → paid at (Date)
9. paid by stays manual
10. No match or low confidence → Needs Review
```

## Step 1: Invoice Upload Gateway

```
Finance user logs in → Upload Invoices page → batch upload → files stored → status Pending
```

1. Finance authenticates into the internal tool (no public access to uploaded invoices).

2. User opens **Upload Invoices** and drops one or many files:
   - PDF, JPG, JPEG, PNG, scanned PDF
   - Excel optional in a later phase

3. Frontend sends `multipart/form-data` to `POST /api/invoices/upload`.

4. Backend stores each file and creates `uploaded_files` metadata:

   ```json
   {
     "original_filename": "adia_invoice_may.pdf",
     "uploaded_at": "2026-05-20T10:00:00Z",
     "uploaded_by": "finance_user_1",
     "processing_status": "Pending",
     "review_status": null
   }
   ```

5. UI shows per-file status: **Pending → Processing → Processed / Failed**.

> **Rule:** `paid at (Date)` is **not** set during invoice upload — only bank matching fills it.

## Step 2: AI Invoice Extraction (DECISION POINT)

```
POST /api/invoices/upload (async or follow-up job) → OCR/text → OpenAI → structured JSON → invoices row
  ✅ High confidence (≥90%) → Auto-save, `review_status = approved`
  ⚠️ Medium (70–89%) → Save, mark Needs Review
  ❌ Low (<70%) or missing critical fields → Needs Review, do not treat as finalized
```

1. **`invoice_extraction_service`** reads the file (Vision for images/PDF pages, or text from OCR).

2. AI returns **structured JSON** (not prose):

   ```json
   {
     "invoice_date": "2026-05-15",
     "name_of_company": "ADIA Group",
     "address_of_company": "Prishtina, Kosovo",
     "invoice_number": "26063",
     "amount": 220.66,
     "currency": "EUR",
     "account_details": "IBAN...",
     "internal_note_description": "Water supply invoice",
     "client_employee_related": "",
     "category": "Utilities",
     "confidence_score": 0.94,
     "needs_review": false
   }
   ```

3. **`ai_validation_service`** flags review when:
   - Invoice number missing
   - Amount missing
   - Company name unclear
   - Invoice date unclear
   - `confidence_score` below threshold

4. Backend maps fields to **Purchase Invoices Database** columns and inserts `invoices`:

   | AI field | Excel / DB column |
   |---|---|
   | `invoice_date` | Invoice Date |
   | `name_of_company` | Name of Company |
   | `address_of_company` | Adress of Company |
   | `invoice_number` | Invoice Number |
   | `invoice_number` → `normalize_invoice_number()` | `invoice_number_normalized` *(DB only — matching key)* |
   | `amount` | Amount |
   | `account_details` | Account Details |
   | `internal_note_description` | Internal Note/Description |
   | `client_employee_related` | Client / Employee Related |
   | — | paid at (Date) *(null until Step 5)* |
   | — | paid by *(null — manual)* |
   | — | Fixed/Not fixed *(manual)* |
   | `category` | Category |

5. Audit log entry: extraction run, confidence, source file id.

**→ High confidence:** visible in Purchase Invoices Table, ready for bank matching.  
**→ Needs review:** appears in **Manual Review Queue** (Step 6).

## Step 3: Bank Statement Upload & Parse

```
Upload Bank Statement Excel → openpyxl read → header detect → bank_transactions rows
```

1. Finance uploads **Bank Statement Excel** via `POST /api/bank-statements/upload`.

2. **`bank_statement_service`** identifies header row (e.g. row **12** in ProCredit `.xls`), imports transaction rows only (e.g. **13–96**; skip summary/footer from row **97**) — see [10. Excel-File-Formats.md](./10. Excel-File-Formats.md).

3. Reads columns:

   | Bank column | Stored as |
   |---|---|
   | Data / Date | `transaction_date` |
   | Vlera e debituar / Debited amount | `debited_amount` |
   | Vlera e kredituar / Credited amount | `credited_amount` |
   | Tipi i veprimit / Type of transaction | `transaction_type` |
   | Komenti / Comment | `comment` |

4. Each row persisted:

   ```json
   {
     "transaction_date": "2026-02-25",
     "debited_amount": 772.39,
     "credited_amount": null,
     "transaction_type": "Transfer",
     "comment": "Payment for invoices 26063, 26060, 260658",
     "detected_invoice_numbers": []
   }
   ```

5. **`extract_invoice_numbers(comment)`** runs on every comment and stores `detected_invoice_numbers`.

   Example:

   ```
   Comment: "Payment for invoices 26063, 26060, 260658"
   Detected: ["26063", "26060", "260658"]
   ```

6. UI **Bank Transactions Table** shows date, amounts, comment, and detected numbers.

> **No bank API** — only uploaded Excel files.

## Step 4: Reconciliation / Matching (CORE DECISION POINT)

```
POST /api/reconciliation/run → normalize numbers → exact match → update paid_at_date OR review task
```

### 4.1 Normalization

Before compare, normalize both sides:

| Raw | Normalized |
|---|---|
| `26.063` | `26063` |
| `INV 2026 001` | `INV2026001` |
| `invoice-26063` | `26063` |

### 4.2 Exact match (MVP)

```
for transaction in bank_statement_rows:
    comment = transaction["comment"]
    transaction_date = transaction["date"]
    invoice_numbers = extract_invoice_numbers(comment)

    for invoice_number in invoice_numbers:
        matched_invoice = find_by_normalized_number(normalize_invoice_number(invoice_number))

        if matched_invoice:
            matched_invoice["paid_at_date"] = transaction_date
            matched_invoice["match_status"] = "Matched"
            create invoice_payment_match record
        else:
            create_review_task(transaction, invoice_number)
```

**Example:**

| Bank comment | Detected | DB Invoice Number | Result |
|---|---|---|---|
| Payment for invoice 26063 | 26063 | 26063 | Match → `paid_at_date` = bank `Data / Date` |

### 4.3 Multiple invoice numbers in one comment

One debit can pay several invoices:

```
Comment: Payment for invoices 26063, 26060, 260658
Date: 25.02.2026
```

→ Update **all** matching invoice rows with the **same** `paid_at_date`.

### 4.4 Batch payment (MVP rule)

| Situation | Action |
|---|---|
| Invoice numbers visible in comment | Match each exact number |
| Numbers not visible in comment | **Do not guess** → Needs Review |
| Advanced (post-MVP) | Suggest batch match via amount sum + supplier + date range |

### 4.5 No match

Mark as **Needs Review**. Common reasons:

- Invoice not uploaded yet
- Number formatted differently in invoice vs bank
- Comment has no invoice reference
- Manual finance case

### 4.6 paid by

**Never** auto-filled. Finance enters **paid by** manually in the table or Excel export.

## Step 5: Paid Date Update (Output to Finance)

Only **`paid at (Date)`** is automated from matching.

```
bank_transactions.transaction_date  →  invoices.paid_at_date
```

Match record:

```json
{
  "invoice_id": 10,
  "bank_transaction_id": 5,
  "invoice_number": "26063",
  "match_type": "invoice_number",
  "match_confidence": 0.98,
  "status": "Matched",
  "paid_at_date": "2026-02-25"
}
```

UI **Matching Results** shows:

- Matched invoices
- Unmatched invoices (no payment date)
- Unmatched bank transactions
- Needs Review cases
- Batch payment candidates (when detectable)

## Step 6: Manual Review Queue

Review items appear when:

| Reason | Source step |
|---|---|
| Missing invoice number | Step 2 |
| Low AI confidence | Step 2 |
| No bank statement match | Step 4 |
| Multiple possible matches | Step 4 |
| Amount mismatch | Step 4 |
| Unclear scanned invoice | Step 2 |

Each queue item shows:

- Original invoice file preview
- AI extracted fields (editable)
- Bank transaction candidate (if applicable)
- Reason for review
- **Approve** / **Reject** / **Save correction**

APIs:

- `POST /api/invoices/{id}/approve`
- `POST /api/reconciliation/approve-match`
- `POST /api/reconciliation/reject-match`

All actions logged in `audit_logs` — see [11. Security & Compliance.md](./11. Security & Compliance.md).

## Operating Principles (for implementation)

1. **AI can extract and suggest** — not replace Finance for ambiguous data.  
2. **System auto-fills `paid at (Date)` only on clear invoice-number match** in bank comment.  
3. **`paid by` is always manual.**  
4. **Never guess** invoice numbers or amounts when comment or confidence is weak.  
5. **Core matching rule:** `Komenti / Comment` → invoice number → `Invoice Number` column → `Data / Date` → `paid at (Date)`.

---

## Definition of Done (MVP)

- [ ] Finance uploads invoice files; AI extracts structured data into the invoice table  
- [ ] Finance uploads Bank Statement Excel; system reads Comment column  
- [ ] System detects and normalizes invoice numbers from comments  
- [ ] System matches to Purchase Invoices Database and fills **paid at (Date)**  
- [ ] **paid by** remains manual  
- [ ] Unmatched / low-confidence cases appear in **Needs Review**  
- [ ] Finance exports updated Purchase Invoices Excel with correct columns  
- [ ] Authentication and RBAC enforced — [11. Security & Compliance.md](./11. Security & Compliance.md)
