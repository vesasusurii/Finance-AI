# Matching & Normalization Technology

> **Principle:** Match on **normalized invoice numbers** only. Auto-fill **`paid at (Date)`** when the link is unambiguous. If the comment does not clearly contain invoice numbers, **do not guess** ‚Äî send to **Needs Review**.

> **Core rule:** `Komenti / Comment` ‚Üí detect numbers ‚Üí normalize ‚Üí lookup `Invoice Number` in Purchase Invoices Database ‚Üí copy `Data / Date` ‚Üí `paid at (Date)`. **`paid by` is never auto-filled.**

## Normalization Specification

**Goal:** Two strings that refer to the same invoice must produce the **same normalized key** before database comparison.

### Algorithm (implement in `normalization.py`)

```python
import re

def normalize_invoice_number(raw: str | None) -> str | None:
    """
    Canonical form for matching. Returns None if input is empty or invalid.
    OCR invoice_number and bank comment candidates MUST both pass through this function.
    """
    if not raw or not str(raw).strip():
        return None

    raw_stripped = str(raw).strip()

    # 1. Slash serial refs (sample_01): "1/2026/0048" -> "120260048"
    if "/" in raw_stripped:
        parts = [p.strip() for p in raw_stripped.split("/") if p.strip()]
        if parts and all(p.isdigit() for p in parts):
            s = "".join(parts)
        else:
            s = re.sub(r"[^A-Z0-9]", "", raw_stripped.upper())
    else:
        s = re.sub(r"\s+", "", raw_stripped.upper())

    # 2. Common prefixes (Albanian + English)
    prefixes = (
        r"^(FATURA|INVOICE|INV|REF|NR|NO|NUM|PAYMENTFORINVOICE(S)?|PAGESA|PAGESAPARFATURES)"
    )
    s = re.sub(prefixes, "", s, flags=re.IGNORECASE)

    # 3. Separators: "INV-2026-001" -> "INV2026001", "26.063" -> "26063"
    if re.fullmatch(r"[\d.,]+", s.replace(",", ".")):
        s = s.replace(",", "").replace(".", "")
    else:
        s = re.sub(r"[^A-Z0-9]", "", s)

    # 4. Reject tax / client registration IDs mistaken for invoice numbers
    if is_tax_or_client_id(s):
        return None

    # 5. Leading zeros: optional ‚Äî document decision (see Pending)
    # s = s.lstrip("0") or "0"   # only for pure numeric keys

    if len(s) < int(os.getenv("MATCH_MIN_DIGIT_LENGTH", "4")) and s.isdigit():
        return None  # too short ‚Äî likely false positive from amount/date

    return s if s else None


def is_tax_or_client_id(s: str) -> bool:
    """Block NUI/UNI/NRF/VAT-style IDs from matching (see OCR-Technology.md)."""
    if s.isdigit() and len(s) == 9 and s.startswith(("811", "330")):
        return True
    if s.isdigit() and len(s) >= 8 and s.startswith("330"):
        return True
    return False
```

## Detection: `extract_invoice_numbers(comment)`

**Module:** `utils/invoice_number_parser.py`  
**When:** After bank Excel import; optionally re-run before reconciliation.

### Regex strategy (MVP ‚Äî no AI required)

Use **multiple patterns**, merge candidates, then normalize and dedupe.

```python
PATTERNS = [
    # Explicit labels (EN + SQ) ‚Äî includes slash refs: 1/2026/0048
    r"(?:invoice|fatura|inv|ref|nr\.?|pagesa)\s*[:#]?\s*([A-Z0-9][\w./-]{2,})",
    # Slash serial without label: 1/2026/0048
    r"\b(\d{1,4}/\d{4}/\d{2,6})\b",
    # "invoice(s) 26063, 26060" or "fature 10210"
    r"(?:invoices?|faturat?|fatures?)\s+([\d\s,.-/]+)",
    # Alphanumeric SaaS-style: 3807F638-0011
    r"\b([A-Z0-9]{4,}[-/][A-Z0-9]{2,})\b",
    # Standalone numeric (4+ digits) ‚Äî apply only after label patterns; filter tax IDs
    r"\b(\d{4,})\b",
]

def extract_invoice_numbers(comment: str) -> list[str]:
    if not comment or not comment.strip():
        return []
    candidates: list[str] = []
    for pattern in PATTERNS:
        for m in re.finditer(pattern, comment, re.IGNORECASE):
            chunk = m.group(1) if m.lastindex else m.group(0)
            for part in re.split(r"[,;]\s*", chunk):
                norm = normalize_invoice_number(part.strip())
                if norm and norm not in candidates and not is_tax_or_client_id(norm):
                    candidates.append(norm)
    return candidates
```

### Optional AI assist (post-MVP)

If regex returns `[]` but comment clearly references payment, **do not** auto-call LLM in MVP. Later: LLM suggests candidates with `match_confidence < 1.0` ‚Üí always `needs_review`.

## Bank Transaction Date Normalization

Matching copies **`Data / Date`** ‚Üí `paid_at_date`. Parse consistently in `bank_statement_service`.

| Input format | Example | Stored |
|---|---|---|
| `DD.MM.YYYY` | `25.02.2026` | `2026-02-25` |
| `YYYY-MM-DD` | `2026-02-25` | `2026-02-25` |
| Excel serial | numeric cell | Convert via `openpyxl` date mode |

If date parse fails ‚Üí `bank_transactions.needs_review = true`; do not match until fixed.

### Entry point

```python
async def run_reconciliation(
    bank_statement_id: int,
    *,
    dry_run: bool = False,
    user_id: int,
) -> ReconciliationResult:
    ...
```

### Per-transaction logic

```python
for tx in bank_transactions_for_statement(bank_statement_id):
    numbers = tx.detected_invoice_numbers or extract_invoice_numbers(tx.comment)
    if not numbers:
        mark_unmatched_transaction(tx, reason="no_invoice_numbers_in_comment")
        continue

    for norm in numbers:
        invoices = find_invoices_by_normalized_number(norm)  # 0, 1, or many

        if len(invoices) == 0:
            create_review_task(
                type="unmatched_bank_reference",
                bank_transaction_id=tx.id,
                invoice_number_candidate=norm,
            )
        elif len(invoices) > 1:
            create_review_task(
                type="multiple_matches",
                bank_transaction_id=tx.id,
                invoice_ids=[i.id for i in invoices],
                invoice_number_candidate=norm,
            )
        else:
            apply_match(
                invoice=invoices[0],
                transaction=tx,
                match_type="invoice_number",
                match_confidence=1.0,
                dry_run=dry_run,
            )
```

### `apply_match` rules

| Field | Behavior |
|---|---|
| `invoices.paid_at_date` | Set to `tx.transaction_date` (if not `dry_run`) |
| `invoices.match_status` | `matched` |
| `invoices.paid_by` | **Never set** |
| `invoice_payment_matches` | Insert row; unique `(invoice_id, bank_transaction_id)` |
| Already paid | If `paid_at_date` already set and differs ‚Üí `needs_review` (`paid_date_conflict`) |

### Match confidence

| Situation | `match_confidence` |
|---|---|
| Exact normalized number, single invoice row | `1.0` |
| Post-MVP amount also matches debited | `0.95` (still can require approval) |
| LLM-suggested number (future) | `‚â§ 0.85` ‚Üí always review |

## Matching Scenarios (decision matrix)

| ID | Scenario | Action |
|---|---|---|
| **A** | One number in comment, one invoice in DB | Auto-set `paid_at_date` (MVP) |
| **B** | Multiple numbers in one comment | Each number matched independently; same `transaction_date` for all |
| **C** | Number in comment, zero invoices in DB | `review_task`: `unmatched_bank_reference` |
| **D** | Number in comment, 2+ invoices same normalized key | `review_task`: `multiple_matches` ‚Äî no auto-update |
| **E** | Empty comment or no numbers detected | Unmatched bank transaction; no invoice updates |
| **F** | Numbers visible but amount mismatch (post-MVP) | Match allowed if configured; else `amount_mismatch` review |
| **G** | Comment without numbers, batch amount equals sum of unpaid (post-MVP) | Suggest only ‚Äî **never auto-apply** in MVP |

### Scenario B example

```
Comment: Payment for invoices 26063, 26060, 260658
Date: 25.02.2026
Debited: 772.39

‚Üí paid_at_date = 2026-02-25 for invoice 26063, 26060, 260658 (if each exists uniquely)
‚Üí three invoice_payment_matches rows pointing to same bank_transaction_id
```

### Scenario D example

```
Two invoice rows both have invoice_number_normalized = "26063"
‚Üí do NOT pick the newer row
‚Üí review queue shows both candidates + bank transaction preview
```

## Idempotency & Re-runs

| Case | Behavior |
|---|---|
| Same `bank_statement_id` reconciled twice | Upsert matches; do not duplicate `invoice_payment_matches` |
| `paid_at_date` already correct | No-op; log `reconciliation_idempotent` |
| `paid_at_date` differs from new run | Create `paid_date_conflict` review ‚Äî do not overwrite silently |
| Invoice deleted after match | Soft-delete matches or cascade per product rule |

8. **Alphanumeric IDs:** Keep full `3807F6380011` / `INV2026001` vs numeric tail only?
