# Excel & File Formats

> **Principle:** You **upload invoices as PDFs and images**, not as Excel. **OCR extracts** the fields and **stores** them in the database. When you need a spreadsheet, you **download** the **Purchase Invoices Database** as **`.xlsx`** or **`.xls`**.

> **Bank statements are the opposite direction:** Finance **uploads** bank data as Excel; the app reads it for matching.

**Related docs:** [2. Tech Stack.md](./2. Tech Stack.md) · [3. Technical Workflow 1.md](./3. Technical Workflow 1.md) · [8. OCR-Technology.md](./8. OCR-Technology.md) · [9. Matching-Normalization.md](./9. Matching-Normalization.md) · [11. Security & Compliance.md](./11. Security & Compliance.md)

## How files move (Finance view)

| Step | What you do | Format | What the system does |
|---|---|---|---|
| **1. Upload invoices** | Drag PDF, scanned PDF, JPG, PNG | **Not Excel** | OCR reads files → saves rows in `invoices` table |
| **2. Review (optional)** | Edit in the app table | — | Updates stored data |
| **3. Upload bank statement** | Upload bank export | **Excel `.xlsx` / `.xls`** | Parses transactions → matching can fill `paid at (Date)` |
| **4. Download invoice data** | Export / Download button | **Excel `.xlsx` or `.xls`** | Builds workbook from DB — same columns as Purchase Invoices Database |

```
  UPLOAD (invoices)              STORE                    DOWNLOAD
  ─────────────────              ─────                    ────────

  PDF ─────┐
  JPG ─────┼──►  OCR extraction  ──►  PostgreSQL         ──►  Purchase_Invoices.xlsx
  PNG ─────┤      (Vision API)         (invoices rows)         Purchase_Invoices.xls
  Scan PDF ┘

  Bank_Statement.xlsx  ──►  excel import  ──►  bank_transactions  ──►  (updates paid at Date in DB)
                                                                                    │
                                                                                    └──► included in next Excel download
```

**You do not need to type invoice data into Excel manually.** Upload documents → OCR fills the system → download Excel when ready.

This is the spreadsheet Finance downloads after OCR has stored invoice rows. It is **not** the upload format for invoices — it is the **export** format.

|---|---|---|---|---|
| 1 | Invoice Date | `invoice_date` | Date `YYYY-MM-DD` or locale display | AI (OCR) |
| 2 | Name of Company | `name_of_company` | Text | AI (OCR) |
| 3 | Adress of Company | `address_of_company` | Text | AI (OCR) — **keep spelling `Adress`** for compatibility |
| 4 | Invoice Number | `invoice_number` | Text (display value) | AI (OCR) |

| 6 | Account Details | `account_details` | Text | AI (OCR) |
| 7 | Internal Note/Description | `internal_note_description` | Text | AI (OCR) |
| 8 | Client / Employee Related | `client_employee_related` | Text | AI suggestion or manual |
| 9 | paid at (Date) | `paid_at_date` | Date | **Matching only** — bank statement |
| 10 | paid by | `paid_by` | Text | **Manual** — Finance only |
| 11 | Fixed/Not fixed | `fixed_status` | Text / dropdown | Manual or dropdown |
| 12 | Category | `category` | Text | AI suggestion + manual correction |

### Rules (critical)

| Column | Rule |
|---|---|
| **paid at (Date)** | **Never** set from OCR. Only from bank reconciliation ([9. Matching-Normalization.md](./9. Matching-Normalization.md)). |
| **paid by** | **Never** auto-filled by matching or OCR. |
| **Invoice Number** | Store display value in Excel; use `invoice_number_normalized` in DB for matching (not exported in official file unless approved). |
| **Amount / currency** | `amount` exports to Excel; `currency` (EUR, USD, etc.) is stored in DB from OCR but **not** in the official 12 columns until Finance approves — see §12 pending #7. |

## 2. Bank Statement (Excel upload)

Uploaded separately after invoices exist in the system. One file = one `bank_statement_id` (metadata + many `bank_transactions`).

**Reference file analyzed:** `Bank Statement 02.2026.xls` (ProCredit-style export, `.xls`).

### Row layout (reference file — 1-based Excel rows)

| Excel rows | Content | Import? |
|---|---|---|
| **1–11** | Account metadata (account number, account name, date range, opening/closing balance) | **No** — read only for UI preview / audit |
| **12** | **Column headers** | **No** — used to map column names only |
| **13–96** | **Transaction rows** (84 rows in Feb 2026 sample) | **Yes** |
| **97–100** | `Përmbledhje/Summary`, blank rows, legal/disclaimer text | **No — never import** |

```
Row 1-11   Account info (Numri i llogarisë, From/To date, balances)
Row 12     HEADER ROW  ← find_header_row() stops here
Row 13     First transaction
   ...
Row 96     Last transaction
Row 97     Përmbledhje/Summary  ← STOP import (exclude through last row)
Row 98-100 Footer / legal text
```

**Rule:** Do **not** import any row from **97 through the last row** (summary + legal footer). Do **not** treat rows **1–12** as transactions. Only **rows 13–96** (in this file) are payment lines.

> **Clarification:** In `Bank Statement 02.2026.xls`, row **13** is the **first payment**, not the start of the footer. Excluding “row 13 through the last row” would remove all transactions. The footer begins at row **97** (`Përmbledhje/Summary`). If a different cutoff was intended, confirm with Finance.

> For other exports: stop at the first row where `Data / Date` contains `Përmbledhje` or `Summary`, or where the row is clearly not a transaction.

### Preamble fields (rows 1–11 — metadata only)

| Label (bilingual) | Example value | Stored as |
|---|---|---|
| Numri i llogarisë / Account number | `XK051110343587000119` | `bank_statements.account_iban` (optional) |
| Emri i llogarisë / Account name | (account holder) | `bank_statements.account_name` (optional) |
| Nga data / From date | `01.02.2026` | `bank_statements.period_from` |
| Ne datën / To date | `28.02.2026` | `bank_statements.period_to` |
| Gjendja fillestare / Previous balance | `31,631.10` | `bank_statements.opening_balance` |
| Gjendja e fundit / Final balance | `35,227.55` | `bank_statements.closing_balance` |

### Column headers (row 12)

Headers may include line breaks inside cells (e.g. `Nr /\nNo`). Normalize by lowercasing and collapsing whitespace before matching.

| Bank header (in file) | DB field | Usage |
|---|---|---|
| **Nr / No** | `row_number` | Sequence — optional |
| **Nr. Transferit / Transfer No** | `transfer_no` | Bank reference — optional |
| **Data / Date** | `transaction_date` | → `paid at (Date)` on match |
| **Vlera e debituar / Debited amount** | `debited_amount` | Outgoing payment (primary for invoice matching) |
| **Vlera e kredituar / Credited amount** | `credited_amount` | Incoming funds |
| **Bilanci / Balance** | `balance_after` | Running balance — optional |
| **Tipi i veprimit / Type of transaction** | `transaction_type` | e.g. E-Pagesë fature, E-Transfertë |
| **Komenti / Comment** | `comment` | **Invoice number detection** — required |

**Most important column for matching:** `Komenti / Comment` — see [9. Matching-Normalization.md](./9. Matching-Normalization.md).

### Comment field patterns (from reference file)

Invoice references appear inside `Komenti / Comment`, often after semicolons:

| Pattern in comment | Example snippet | Normalized target |
|---|---|---|
| `Inv 1/2026/0048` | slash serial | `120260048` |
| `Inv 26/0863` | multi-invoice list | `260863`, etc. |
| `INV 5945698` | numeric | `5945698` |
| `Inv 101/26` | slash form | `10126` or preserve per [9. Matching-Normalization.md](./9. Matching-Normalization.md) |
| `Invoice 190053494260100J` | long numeric | extract full id |

POS/card lines (e.g. `OPENAI` in comment) may have **no** invoice number — leave `detected_invoice_numbers` empty; do not match.

### Parsed row JSON (after import)

```json
{
  "transaction_date": "2026-02-25",
  "debited_amount": 772.31,
  "credited_amount": 0,
  "transaction_type": "E -Transfertë brenda bankës",
  "transfer_no": "737609214",
  "comment": "… Inv 26/0863, 26/0860, 26/0658 …",
  "detected_invoice_numbers": ["260863", "260860", "260658"]
}
```

### Import algorithm

```python
HEADER_ROW_1BASED = 12          # scan to find if bank changes layout
FIRST_DATA_ROW_1BASED = 13      # first transaction in reference file
STOP_MARKERS = ("përmbledhje", "summary", "fsdk", "deposit insurance")

def iter_transaction_rows(sheet):
    header_idx = find_header_row(sheet)  # expect 1-based row 12
    for row_num in range(FIRST_DATA_ROW_1BASED, sheet.max_row + 1):
        row = sheet[row_num]
        if is_summary_or_footer(row, STOP_MARKERS):
            break  # do not read row 97 → last row
        if is_empty_transaction(row):
            continue
        yield parse_transaction_row(row)
```

| Failure | User message |
|---|---|
| No header row found | “Could not find bank statement headers (Data / Date, Komenti / Comment).” |
| Missing `Komenti / Comment` | “Comment column is required for invoice matching.” |
| No rows in range 13–summary | “No transaction rows found.” |
| Summary row reached | Stop import silently; log `stopped_at_row=97` |

### Cell parsing rules

| Field | Excel cell types | Normalized storage |
|---|---|---|
| `transaction_date` | `DD.MM.YYYY`, `DD/MM/YYYY`, Excel serial date | ISO `YYYY-MM-DD` |
| `debited_amount` / `credited_amount` | Number, or text with `,` decimal | `Decimal` / float, 2 dp |
| `comment` | Text | Trim whitespace; preserve original in `comment_raw` |

If date parse fails → mark row `needs_review`; do not run match on that row until fixed.

### Matching use of amounts (MVP)

| Field | MVP behavior |
|---|---|
| `debited_amount` | Primary for outgoing invoice payments |
| `credited_amount` | Store; matching by invoice number does not require amount in MVP |
| Amount vs invoice total | Post-MVP optional check ([9. Matching-Normalization.md](./9. Matching-Normalization.md) scenario F) |

## 5. Technical Implementation

### Libraries

| Package | Use |
|---|---|
| `openpyxl` | Read/write `.xlsx` (Bank Statement + export) |
| `pandas` (optional) | Bulk read alternative; team picks one primary path |
| `xlrd` (optional) | Legacy `.xls` only if Finance still sends old format |

Prefer **one** read stack in production to avoid behavior drift.

### Workbook conventions

| Topic | Convention |
|---|---|
| Sheet index | First sheet unless Finance specifies sheet name in config |
| Header row | Dynamic — not always row 1 |
| Formulas | Import: read **computed values** (`data_only=True` in openpyxl) |
| Blank rows | Skip |
| Thousands separators | Parse `1.931,78` vs `1,931.78` based on locale heuristics or bank format config |

### Export formatting

| Column | Excel format |
|---|---|
| Invoice Date, paid at (Date) | Date `dd.mm.yyyy` or ISO per Finance preference |
| Amount | Number `#,##0.00` |
| Invoice Number | **Text** — prevent scientific notation on long IDs (e.g. `132018959018`) |
| Long numeric IDs | Prefix cell with `'` or set number format `@` (text) |

## Phase 5 — Monthly Report (post-MVP)

`GET /api/export/monthly-report` — separate from the Purchase Invoices Database export.

| Output | Source |
|---|---|
| Total invoice amount for month | `SUM(invoices.amount)` filtered by `invoice_date` or `paid_at_date` (confirm with Finance) |

| Unmatched invoices | `match_status = unmatched` |
| Unmatched bank transactions | `reconciliation_status` pending / needs_review |
| Category breakdown | `GROUP BY category` |

**Format:** Excel or PDF — pending Finance decision. **Auth:** same as PI export ([11. Security & Compliance.md](./11. Security & Compliance.md)). **Audit:** log `export_run` with report type `monthly`.

## 13. Definition of Done (Excel & Files)

- [ ] Bank Statement upload parses `Data / Date`, debited/credited amounts, `Komenti / Comment`  
- [ ] Rows **1–12** skipped (metadata + header); rows **97–end** skipped (summary/footer)  
- [ ] Only transaction rows imported (e.g. **13–96** in `Bank Statement 02.2026.xls`)  
- [ ] Header row detection finds row **12** in ProCredit-style `.xls`  
- [ ] Export produces exactly the **12 official columns** in correct order and spelling (`Adress`)  
- [ ] `paid at (Date)` only populated from matching, empty after OCR-only insert  
- [ ] `paid by` never auto-populated  
- [ ] Long `invoice_number` values export as text, not scientific notation  
- [ ] Invalid Excel returns clear 4xx errors  
- [ ] Original bank upload retained for audit  
- [ ] Invoice upload accepts PDF/images only; OCR persists to DB without Excel upload  
- [ ] `GET /api/export/purchase-invoices-excel` downloads valid `.xlsx` and optionally `.xls`

## Architecture Summary

| Capability | Implementation |
|---|---|
| Invoice upload | **PDF / images** → OCR → **PostgreSQL** |
| Invoice download | DB → **Purchase Invoices Database** `.xlsx` or `.xls` |
| Bank ingest | Upload **Bank Statement** `.xlsx` / `.xls` → `bank_transactions` |
| Excel library | `openpyxl` (`.xlsx`); optional `xlwt` or similar (`.xls` export) |
| Column contract | Fixed 12 columns on every download |
| Matching output | Updates `paid_at_date` in DB → appears on next Excel download |
