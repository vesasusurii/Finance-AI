## Component-by-Component Breakdown

---

### 1. AI — Invoice Extraction & Validation

**Provider policy:** **OpenAI Vision** is primary; **Google Document AI** only when OpenAI is unavailable. Details: [8. OCR-Technology.md](./8. OCR-Technology.md).

**Recommended MVP flow (from project spec):**

1. **OCR / document parser** extracts raw text from PDF, scanned PDF, or photo
2. **OpenAI** validates and converts the result to clean structured JSON
3. **Backend** saves JSON into PostgreSQL (and optionally exports to Excel)

**Tech options:**

| Option | Role | Good for |
|---|---|---|
| **OpenAI Vision / GPT** | Read invoice images/PDFs, return structured JSON | Different layouts, photos, scanned invoices, flexible fields |
| **Google Document AI Invoice Parser** *(optional)* | Invoice-specific OCR + structured fields | Production reliability, standard invoice layouts |

**Required extraction output (structured JSON — not free text):**

```json
{
  "invoice_date": "",
  "name_of_company": "",
  "address_of_company": "",
  "invoice_number": "",
  "amount": "",
  "currency": "",
  "account_details": "",
  "internal_note_description": "",
  "client_employee_related": "",
  "category": "",
  "confidence_score": 0.0,
  "needs_review": true
}
```

**Confidence rules:**

| Confidence | Action |
|---|---|
| 90–100% | Auto-save |
| 70–89% | Save but mark for review |
| Below 70% | Do not finalize — require manual review |

### 2. Backend — Python + FastAPI

All business logic runs in a Python + FastAPI server. Fully owned, version controlled, deployable to Railway, Render, VPS, etc.

**Core modules (from project spec):**

| Module | Responsibility |
|---|---|
| `invoice_extraction_service` | OCR + AI extraction, confidence scoring, validation |
| `bank_statement_service` | Read Bank Statement Excel, parse rows, store transactions |
| `matching_service` | Invoice number detection, normalization, exact match, paid date update |
| `excel_service` | Read/write Purchase Invoices Database Excel format |
| `ai_validation_service` | Post-extraction validation and review flags |

**API routes:**

| Route | Purpose |
|---|---|
| `POST /api/auth/login` | Issue JWT/session (Finance users only) |
| `GET /health` | Health check (no auth) |
| `POST /api/invoices/upload` | Batch upload invoice files (PDF, JPG, PNG, scanned PDF) |
| `GET /api/invoices` | List invoices with filters |
| `GET /api/invoices/{id}` | Single invoice detail |
| `PUT /api/invoices/{id}` | Manual edit by Finance |
| `DELETE /api/invoices/{id}` | Remove invoice row |
| `POST /api/invoices/{id}/approve` | Approve after review |
| `POST /api/bank-statements/upload` | Upload Bank Statement Excel |
| `GET /api/bank-statements` | List uploaded statements |
| `GET /api/bank-transactions` | Transaction rows with detected invoice numbers |
| `POST /api/reconciliation/run` | Run matching: comment → invoice number → `paid_at_date` |
| `GET /api/reconciliation/results` | Matched / unmatched / needs review |
| `POST /api/reconciliation/approve-match` | Finance approves a suggested match |
| `POST /api/reconciliation/reject-match` | Finance rejects a match |
| `GET /api/export/purchase-invoices-excel` | Export Purchase Invoices Database Excel |
| `GET /api/export/monthly-report` | Phase 5 reporting |
| `GET /api/files/{id}` | Authenticated download of invoice PDF/image ([11. Security & Compliance.md](./11. Security & Compliance.md)) |
| `GET /api/review/tasks` | List manual review queue items |

Full request/response contracts: **5. API Specification 1.md**

**Invoice number parser (core matching helper):**

```python
def extract_invoice_numbers(comment: str) -> list[str]:
    """
    Detect possible invoice numbers from bank statement Komenti / Comment.
    Return normalized candidates (e.g. "26.063" -> "26063").
    """
```

Supported comment patterns: `INV-2026-001`, `2026/001`, `26063`, `Invoice 26063`, `Fatura 26063`, `Ref: 26063`, `Payment for invoice 26063`, batch strings like `26063, 26060, 260658`.

### 3. Database — PostgreSQL

PostgreSQL is the system of record. The **Purchase Invoices Database** Excel is generated/exported from the database — Finance keeps the same column structure on export.

**Why database + Excel export (not Excel-only):**

- Safer concurrent edits
- Audit logs and review queue
- Matching history (`invoice_payment_matches`)
- Easier reconciliation reruns

**Core tables:** `invoices`, `bank_transactions`, `invoice_payment_matches`, `uploaded_files`, `review_tasks`, `audit_logs`, `users`

**Hosting:** Any managed Postgres (Supabase, Neon, Railway) or self-hosted. Backend connects via SQLAlchemy + `asyncpg`.

### 5. Excel Files — Input / Output (not live DB for Finance)

**Purchase Invoices Database** (main output file):

| Column | Filled by |
|---|---|
| Invoice Date | AI |
| Name of Company | AI |
| Adress of Company | AI |
| Invoice Number | AI (+ `invoice_number_normalized` in DB for matching — [9. Matching-Normalization.md](./9. Matching-Normalization.md)) |
| Amount | AI |
| Account Details | AI |
| Internal Note/Description | AI |
| Client / Employee Related | AI suggestion or manual |
| paid at (Date) | **System from bank statement match only** |
| paid by | **Manual by Finance** |
| Fixed/Not fixed | Manual or dropdown |
| Category | AI suggestion + manual correction |

**Bank Statement Excel** (upload only — no bank API):

| Column | Usage |
|---|---|
| Data / Date | Payment date → `paid at (Date)` when matched |
| Vlera e debituar / Debited amount | Amount paid out |
| Vlera e kredituar / Credited amount | Amount received |
| Tipi i veprimit / Type of transaction | Transaction type |
| **Komenti / Comment** | **Invoice number detection (critical)** |

### 6. Matching Engine (our code — no third-party matcher)

**Core rule:**

```
Bank Statement Komenti / Comment
  → detect invoice number(s)
  → find same Invoice Number in Purchase Invoices Database
  → copy Bank Statement Data / Date
  → write to paid at (Date)
```

**MVP matching:**

- Exact match after normalization
- One comment with multiple numbers → update all matching invoice rows with same payment date
- No visible invoice numbers in comment → **Needs Review** (do not guess)
- `paid by` never auto-filled

**Advanced (post-MVP):** amount-combination batch matching when comment lacks all numbers.

## Architecture Summary

| Capability | Implementation |
|---|---|
| Invoice upload | **React** drag-and-drop → **FastAPI** → file storage + `uploaded_files` |
| AI invoice extraction | **OpenAI** (structured JSON + confidence) |
| Purchase Invoices Database | **PostgreSQL `invoices`** → **export** to Excel (same columns) |
| Bank statement ingest | **Uploaded Excel only** — `openpyxl` → `bank_transactions` |
| Invoice number detection | **`invoice_number_parser`** (regex + normalization) |
| Payment date auto-fill | **`matching_service`** — Comment → Invoice Number → `paid_at_date` |
| paid by | **Manual only** — never automated |
| Unclear cases | **`review_tasks`** queue + `needs_review` flags |
| Audit | **`audit_logs`** — extractions, edits, payment date updates |
| Auth | **App authentication + RBAC** (required for financial data) |
| Bank API | **Not used** — uploaded files only |

## Local Development & Deployment

**Default for the team:** Docker Compose (`db` + `backend`; `frontend` with profile `full` when scaffolded).

| Component | Local (Compose) | Production |
|---|---|---|
| PostgreSQL | `postgres:16-alpine` service | Managed Postgres (SSL, backups) |
| Backend | `backend/Dockerfile` + hot reload | Same image, no `--reload` |
| Frontend | Profile `full` or host `npm run dev` | Static build + HTTPS |
| Uploads | Volume `upload_data` | Volume or S3-compatible |
| Secrets | `.env` (not in Git) | Platform secret manager |

Full setup, env vars, migrations, troubleshooting: **12. Deployment & Environments 1.md**
